# WTCSEOUL 품의 보고서 프롬프트 설계실 PRD

## 0. Antigravity 개발 지시 요약

이 서비스는 **AI API를 호출하지 않는 프론트엔드 전용 사이트**다. 사용자가 문서 종류와 업무유형을 선택하고 필요한 작업값을 입력하면, ChatGPT/Claude/사내 AI에 붙여넣을 수 있는 **최종 프롬프트**를 생성한다.

- 핵심 목표: 품의서/협조전/대행예산 기안문/보고서 작성 프롬프트 자동 생성
- API 사용 금지: OpenAI, Claude 등 외부 API 호출 없음
- 배포 목표: Vercel 또는 Cloudflare Pages 정적 배포
- 기술스택: React + Vite + TypeScript + Tailwind CSS
- 저장 방식: 기본은 브라우저 localStorage, 추후 Supabase 연동 가능하도록 데이터 구조 분리
- 기준 자료: 업로드된 예시 기안문 PDF, 최신 규정집, 문서관리요령

---

## 0.1 이번 버전 반영사항

- 품의 작성과 보고서 작성을 별도 메뉴로 분리한다.
- 품의 작성 문서종류는 `기안문`, `협조전`, `기안문(대행예산)`으로 구분한다.
- 사이트는 AI API를 호출하지 않고, 사용자가 Claude/ChatGPT에 붙여넣을 수 있는 최종 실행 프롬프트만 생성한다.
- Gemini식 고급 프롬프트 설계 원칙을 반영해 `입력값 → 구조화 → 연결 → 검증 → 최종 프롬프트` 흐름으로 설계한다.
- 최종 실행 프롬프트 마지막에 업무유형별 추천 첨부파일 표를 반드시 생성하도록 한다.
- 결재선, 보존년한, 문서보안은 자동 확정이 아니라 규정 기반 추천으로 표시한다.

## 1. 제품 개요

### 1.1 제품명

**WTCSEOUL 품의 보고서 프롬프트 설계실**

### 1.2 한 줄 설명

업무유형과 작업값을 입력하면, 회사 양식과 규정 기준에 맞춘 **품의/협조전/보고서 작성용 프롬프트**를 자동으로 만들어주는 사내 업무 도구.

### 1.3 만들고 싶은 이유

현재 품의 작성은 기존 결재문서를 보고 따라 쓰는 방식이 많다. 하지만 예전 문서의 결재선, 보존년한, 문서보안, 첨부자료 구성이 최신 규정과 맞지 않을 수 있다. 따라서 사이트가 사용자의 입력값을 기준으로 다음을 자동 추천해야 한다.

1. 적절한 문서 종류
2. 업무유형별 입력값
3. 제목 패턴
4. 결재선 검토 포인트
5. 문서보안 및 보존년한 추천
6. 첨부파일 추천
7. 최종 AI 작성 프롬프트

1. 품의/협조전 프롬프트 생성
2. 보고서 작성 프롬프트 생성
3. 업무유형별 입력폼 자동 변경
4. 제목 패턴 추천
5. 결재선 검토 포인트 자동 표시
6. 일상감사/검사역 입회 필요 여부 표시
7. 문서보안/보존년한 추천
8. 첨부파일 추천
9. 최종 프롬프트 복사
10. 예시 프롬프트 제공
11. localStorage 기반 최근 작성 내역 저장
12. 규칙/업무유형 데이터 수정 가능 구조

### 2.2 제외 기능

1. AI API 호출
2. 실제 품의서 파일 자동 작성
3. 전자결재 시스템 연동
4. 사내 DB 연동
5. 첨부파일 자동 분석
6. 사용자 로그인
7. 결재권자 최종 확정 기능

### 2.3 중요한 원칙

사이트는 **문서를 직접 작성해주는 도구가 아니라, 문서를 잘 작성하도록 AI에게 지시하는 프롬프트를 만들어주는 도구**다.

따라서 결과물은 다음 형태다.

```text
너는 WTC Seoul 오피스운영팀 실무자야.
아래 입력값을 기준으로 [기안문/협조전/보고서] 초안을 작성해줘.
...
```



### 2.4 프롬프트 생성 공통 원칙

이 사이트는 실제 문서를 직접 작성하는 AI 서비스가 아니라, 사용자가 Claude 또는 ChatGPT에 붙여넣을 수 있는 **최종 실행 프롬프트**를 생성하는 도구다. 재무모델링 프롬프트처럼 복잡한 업무를 `입력값 → 구조화 → 연결 → 검증 → 최종 프롬프트` 흐름으로 설계하되, WTC Seoul 문서작성 업무에 맞춰 아래 원칙을 적용한다.

#### 2.4.1 단순 문장 생성으로 끝내지 않는다

- `품의서 써줘`, `보고서 써줘` 수준의 단순 프롬프트를 생성하지 않는다.
- 문서종류, 업무유형, 입력값, 결재 검토사항, 첨부파일, 보존년한, 문서보안, 작성 형식까지 포함한 실행형 프롬프트를 생성한다.
- 프롬프트 결과는 반드시 `역할 부여 → 입력값 정리 → 문서작성 기준 → 출력 형식 → 첨부파일 추천 → 결재 전 검증` 순서로 구성한다.

#### 2.4.2 원본 자료는 수정하지 않는 원칙을 둔다

- 사용자가 입력한 원자료, 과거 품의 예시, 규정 문구는 원본으로 간주한다.
- 생성 프롬프트 안에는 `원본 자료의 사실관계는 임의로 변경하지 말 것`을 반드시 포함한다.
- 금액, 기간, 면적, 계약조건, 업체명, 입주사명 등은 사용자가 입력한 값만 사용하도록 한다.
- 미입력 값은 임의 추정하지 않고 `직접 입력`으로 표시한다.

#### 2.4.3 입력값과 결과물을 분리한다

- 입력값 영역과 생성 결과 영역을 명확히 분리한다.
- 입력값: 문서종류, 업무유형, 건물명, 호실, 입주사명/업체명, 금액, 기간, 사유, 재원, 관련문서, 첨부파일 등.
- 결과물: 품의 제목, 본문 구조, 표 구성, 첨부파일 추천, 결재 전 확인사항, 문서관리 기준.

#### 2.4.4 가정값에는 출처를 남긴다

- 임대료 단가, 관리비 단가, 보증금, 렌트프리, 계약기간, 출장비, 견적금액 등 주요 입력값에는 출처를 남기도록 안내한다.
- 출처 예시: 임대차계약서(안), 견적서, 관련문서, 규정집, 정산내역서, 출장계획, 내부 정책자료.
- 기준일, 단위, 작성자 정보를 함께 남기도록 한다.

#### 2.4.5 문서종류별 구조를 다르게 적용한다

- 기안문, 협조전, 기안문(대행예산), 보고서는 같은 방식으로 작성하지 않는다.
- 기안문: 결재용 문서로서 제목, 도입문, `- 다 음 -`, 본문, 첨부, 끝 표시를 포함한다.
- 협조전: 타 부서 요청 목적이므로 수신처, 요청사항, 후속 업무, 첨부를 명확히 한다.
- 기안문(대행예산): 예산소요, 재원, 계정명, 기안금액, 예산 검토사항을 포함한다.
- 보고서: 결재용 품의가 아니라 검토·공유용 문서이므로 배경, 현황, 이슈, 검토의견, 향후계획 중심으로 작성한다.

#### 2.4.6 문서관리요령을 반영한다

- 항목 구분은 원칙적으로 `1. → 가. → (1) → (가) → ①` 순서로 안내한다.
- 본문 종료 시 `끝.` 표시를 포함하고, 첨부가 있는 경우 첨부명과 수량을 적은 뒤 `끝.`으로 마무리하도록 한다.
- 보존년한은 기본 3년을 원칙으로 하되, 계약서·회계증빙·대외문서·소송문서 등은 별도 기준으로 추천한다.
- 문서보안은 기본 `해당팀` 기준으로 추천하되, 계약서·소송·인감·입찰·중요 정보가 포함된 문서는 상향 검토하도록 안내한다.

#### 2.4.7 결재선은 자동 확정이 아니라 추천으로 표시한다

- 결재선은 규정 기준에 따라 추천하되, 최종 판단은 사용자가 하도록 한다.
- 금액 기준은 부가세 별도 기준으로 판단하도록 안내한다.
- 일반예산, 임대차계약, 보증금 정산, 시설공사, 협력업체 계약 등 업무유형별 전결 기준을 분리한다.
- 전결권자 판단이 애매한 경우 `규정 재확인 필요`로 표시한다.

#### 2.4.8 검증 항목을 반드시 포함한다

생성 프롬프트 마지막에는 `결재 전 확인사항`을 포함한다. 최소 검증 항목은 다음과 같다.

1. 금액이 부가세 별도/포함 중 무엇인지 확인.
2. 결재선이 업무유형과 금액 기준에 맞는지 확인.
3. 관련문서가 누락되지 않았는지 확인.
4. 첨부파일이 업무유형에 맞게 들어갔는지 확인.
5. 보존년한과 문서보안이 문서 성격에 맞는지 확인.
6. 본문 끝/첨부 끝 표시가 맞는지 확인.
7. 임의로 만든 사실이나 금액이 없는지 확인.

#### 2.4.9 첨부파일 추천을 마지막에 제공한다

- 업무유형과 문서종류에 따라 필요한 첨부파일을 자동 추천한다.
- 필수 첨부와 선택 첨부를 구분한다.
- 첨부파일이 없으면 `첨부 없음`이 아니라 `첨부 필요 여부 확인`으로 표시한다.
- 생성 프롬프트 하단에 `추천 첨부파일`, `필수/권장/조건부`, `추천 사유`, `누락 시 리스크`를 표로 넣도록 지시한다.

#### 2.4.10 결과 해석과 판단은 분리한다

- AI가 결재 가능 여부를 단정하지 않도록 한다.
- 프롬프트는 초안 작성과 체크리스트 제공까지 수행하고, 최종 결재 판단은 사용자가 하도록 안내한다.
- 결재선, 보존년한, 문서보안은 모두 `추천`으로 표시한다.

## 5. 문서 대분류

### 5.1 품의/협조전 작성

전자결재에 올리는 문서용 프롬프트를 생성한다.

#### 문서종류

| 문서종류 | 용도 | 예시 |
|---|---|---|
| 기안문 | 일반 품의, 계약, 공사, 완료조서, 보증금 반환, 입찰 등 | 공사 시행, 임대차계약, 완료조서, 보증금 반환 |
| 협조전 | 타 부서 협조 요청, 직인/인감, 재예치, 자재 불출, 교육비 신청 등 | 사용인감 요청, 보증금 재예치 요청, 천장텍스 불출 요청 |
| 기안문(대행예산) | 대행예산/특별회계/재원 입력이 필요한 문서 | 구매, 출장 시행, 대행예산 공사, 입찰 추진 |

### 5.2 보고서 작성

결재보다는 공유·검토·보고 목적의 문서용 프롬프트를 생성한다.

#### 보고서 유형

| 보고서 유형 | 용도 | 예시 |
|---|---|---|
| 추진안 | 새로운 사업, 제도, 개선안을 추진할 때 | 입주사 앱 추진안, 로비 개선 추진안 |
| 검토보고서 | 여러 대안을 비교·검토할 때 | 비용 비교, 정책 검토, 시설 개선 검토 |
| 결과보고 | 출장, 회의, 점검, 입찰 결과를 보고할 때 | 출장 결과보고, 입찰 결과보고 |
| 현황보고 | 현재 상태, 수치, 이슈를 정리할 때 | GBD 현황, 공실 현황, 민원 현황 |

## 11. 규정 기준 자동 추천 로직

### 11.1 금액 기준 공통 원칙

- 모든 금액 판단은 **부가세 별도 금액** 기준으로 한다.
- 동일 건을 금액 분할하여 결재선, 일상감사, 검사역 입회를 회피하지 않도록 경고한다.

### 11.2 일반예산 품의/지출결의 전결선

| 부가세 별도 금액 | 추천 결재선 |

## 12. 문서관리요령 반영 규칙

### 12.1 항목 번호

최종 프롬프트에는 다음 항목 체계를 지키라고 지시한다.

```text
1. 첫째 항목
가. 둘째 항목
(1) 셋째 항목
(가) 넷째 항목
① 다섯째 항목
```

### 12.2 문서 종결

- 첨부가 없으면 본문 마지막 문장 뒤에 `끝.` 표시
- 첨부가 있으면 실무 예시 기준으로 `(첨부)` 또는 규정형 `붙임` 중 선택 가능
- 기본값은 예시파일에서 많이 쓰는 `(첨부)` 형식으로 한다.

#### 실무형 기본값

```text
(첨부) 1. 견적서 - 1부
      2. 현장사진 - 1부. 끝.
```

#### 규정형 선택값

```text
붙임  1. 견적서 1부
      2. 현장사진 1부.  끝.
```

### 12.3 보존년한 추천

| 문서 성격 | 추천 보존년한 |
|---|---|
| 단순 협조, 교육비, 자재 불출, 단순 요청 | 1년 |
| 일반 기안문, 공사 시행, 완료조서, 보증금 반환 | 3년 |
| 구매, 대행예산, 경비지출 증빙, 일반 계약 관련 | 5년 |
| 대표이사 전결 이상의 만기/해약 계약서, 회계서류 | 10년 |
| 제규정, 이사회/주주총회 회의록, 소송, 중요 인허가, 중요 계약서 | 영구 |

| 회의록 | 필수 | 협의내용 근거 |
| 영수증 | 필수 | 회의비/출장비 정산 근거 |
| 현장사진 | 권장 | 시설점검 결과 확인 |
| 발표자료 | 조건부 | 워크샵/회의 자료 있을 경우 |
| 점검표 | 권장 | 안전/시설 점검 시 |

#### 13.3.13 교육비 신청

| 첨부파일 | 필수 여부 | 추천 사유 |
|---|---|---|
| 자격증취득계획서 또는 교육계획서 | 필수 | 교육 필요성 확인 |
| 교육기관 안내자료 | 권장 | 교육기관 및 과정 확인 |
| 결제영수증 | 필수 | 청구금액 근거 |
| 교육과정 커리큘럼 | 권장 | 직무 관련성 확인 |
| 시험일정 자료 | 조건부 | 자격증 취득 목적 시 |

#### 13.3.14 입찰·계약·용역

| 첨부파일 | 필수 여부 | 추천 사유 |
|---|---|---|
| 입찰 제안요청서(RFP) | 필수 | 과업범위 및 입찰조건 |
| 입찰공고문 | 권장 | 절차 근거 |
| 예가조서 또는 산출내역 | 필수 | 예정가격 근거 |
| 종합평가표 | 제안평가 시 필수 | 정량/정성평가 근거 |
| 제안서 | 입찰 결과 시 필수 | 업체 제안내용 확인 |
| 평가위원 서명자료 | 조건부 | 평가 진행 근거 |
| 입찰 결과보고서 | 결과보고 시 필수 | 선정 결과 근거 |
| 계약서 최종 날인본 | 계약체결 후 필수 | 계약 완료 증빙 |

#### 13.3.15 보고서 작성

| 첨부파일 | 필수 여부 | 추천 사유 |
|---|---|---|
| 원자료 또는 근거자료 | 필수 | 보고 내용의 출처 |
| 수치 산출표 | 권장 | 정량 근거 확인 |
| 사진자료 | 조건부 | 현장/공간 개선 보고 시 |
| 비교표 | 권장 | 현황 vs 개선안 비교 |
| 일정표 | 권장 | 실행계획 포함 시 |
| 비용 산출근거 | 조건부 | 예산/비용 언급 시 |


### 13.4 최종 프롬프트 하단 첨부파일 추천 출력 규칙

생성되는 최종 실행 프롬프트는 마지막에 반드시 다음 지시를 포함한다.

```text
[첨부파일 추천]
아래 입력값과 문서종류, 업무유형을 기준으로 이 문서에 붙이면 좋은 첨부파일을 추천해줘.
추천 첨부파일은 표로 정리하고, 필수/권장/조건부를 구분해줘.
각 첨부파일마다 추천 사유와 누락 시 검토 리스크를 함께 적어줘.
사용자가 이미 보유한 첨부파일과 추가로 확인해야 할 첨부파일을 나눠줘.
규정상 원본 별도 보관이 필요한 계약서, 소송문서, 인허가 문서, 확정일자/내용증명 문서는 별도 보관 필요 항목으로 표시해줘.
```

첨부파일 추천 결과 표는 아래 컬럼을 따른다.

| 구분 | 첨부파일 | 필수 여부 | 추천 사유 | 누락 시 리스크 | 원본보관 필요 여부 |
|---|---|---|---|---|---|
| 예시 | 임대차계약서(안) | 필수 | 계약조건 확인 | 계약조건 검토 불가 | 계약 체결본은 원본보관 필요 |

## 14. 프롬프트 생성 규칙

### 14.1 품의/협조전 최종 프롬프트 템플릿

```text
너는 WTC Seoul 오피스운영팀의 문서작성에 능숙한 실무자야.
아래 입력값을 기준으로 회사 전자결재용 {문서종류} 초안을 작성해줘.

[문서 작성 목적]
- 문서종류: {기안문/협조전/기안문(대행예산)}
- 업무유형: {업무유형}
- 작성 목적: {시행/체결/지급/반환/협조요청/보고}

[입력값]
- 건물명: {건물명}
- 위치/호실: {위치}
- 입주사/업체명: {회사명}
- 업무명: {업무명}
- 금액: {금액}원, 부가세 {별도/포함}
- 기간: {기간}
- 관련문서: {관련문서}
- 시행/요청 사유: {사유}
- 주요내용: {상세내용}
- 재원: {재원}
- 특이사항: {특이사항}

[작성 기준]
1. 제목은 추천 제목 패턴을 참고해 회사 기안문 스타일로 작성한다.
2. 첫 문장은 “○○을 다음과 같이 ○○하고자 합니다.” 형식으로 작성한다.
3. 본문은 “- 다 음 -” 또는 “- 아 래 -” 아래에 정리한다.
4. 항목 번호는 1. → 가. → (1) → (가) → ① 순서를 지킨다.
5. 금액은 천단위 콤마와 단위를 붙이고, 부가세 별도/포함 여부를 명확히 표시한다.
6. 표가 필요한 경우 표로 정리한다.
7. 입력되지 않은 사실은 임의로 만들지 말고 “직접 입력”으로 표시한다.
8. 마지막에는 추천 첨부파일 목록을 포함한다.
9. 첨부가 있는 경우 “(첨부) 1. ○○ - 1부. 끝.” 형식으로 마무리한다.
10. 결재선, 문서보안, 보존년한은 추천값으로만 표시하고 최종 확인 필요 문구를 넣는다.

[추가 검토사항]
- 일상감사 필요 여부: {자동계산 결과}
- 검사역 입회 필요 여부: {자동계산 결과}
- 계약담당팀 합의 필요 여부: {자동계산 결과}
- 문서보안 추천: {추천값}
- 보존년한 추천: {추천값}

[결과물]
1. 추천 제목
2. 본문 초안
3. 표 정리
4. 추천 첨부파일
5. 결재 전 확인사항
```

### 14.2 보고서 최종 프롬프트 템플릿

```text
너는 15년 차 부동산 자산관리 및 오피스 운영 보고서 작성 전문가야.
아래 입력값을 기준으로 WTC Seoul 내부 보고서 초안을 작성해줘.

[보고서 정보]
- 보고서 유형: {추진안/검토보고서/결과보고/현황보고}
- 제목: {제목}
- 주관팀: {주관팀}
- 작성 목적: {작성목적}
- 보고 대상: {보고대상}

[입력값]
- 배경: {배경}
- 현황: {현황}
- 주요 수치: {수치}
- 이슈/문제점: {이슈}
- 검토안/대안: {대안}
- 기대효과: {기대효과}
- 실행계획: {실행계획}
- 비용/예산: {비용}
- 첨부자료: {첨부자료}

[작성 기준]
1. 보고서는 결재용 기안문이 아니라 검토·공유용 문서로 작성한다.
2. 결재선, 문서번호, 보존년한, 문서보안 항목은 넣지 않는다.
3. 도입문 “아래와 같이 ~하고자 합니다.”는 사용하지 않는다.
4. 목차는 사안에 맞게 구성하되, 기본 흐름은 핵심 과제 → 현황 분석 → 검토 의견 → 실행 제안 → 기대효과 순서로 한다.
5. 항목 마커는 ㅇ → - → ㆍ 순서를 기본으로 한다.
6. 수치성 내용은 표로 정리한다.
7. 문장은 짧고 보고용 문체로 작성한다.
8. 마지막은 “(끝)”으로 마무리한다.
9. 입력되지 않은 사실은 임의로 만들지 말고 “직접 입력”으로 표시한다.
10. 마지막에 보고서에 붙이면 좋은 첨부자료를 추천한다.

[결과물]
1. 보고서 제목
2. 보고서 목차
3. 본문 초안
4. 표 정리
5. 첨부자료 추천
6. 후속 품의 전환 시 필요한 확인사항
```

## 22. 수용 기준

### 22.1 기능 기준

- 사용자가 문서종류를 선택할 수 있어야 한다.
- 업무유형 선택 시 입력 필드가 바뀌어야 한다.
- 금액 입력 시 결재선/일상감사/검사역 안내가 표시되어야 한다.
- 최종 프롬프트가 생성되어야 한다.
- 프롬프트 복사 버튼이 동작해야 한다.
- 업무유형별 첨부파일 추천이 표시되어야 한다.
- 보고서 작성은 품의/협조전과 별도 메뉴여야 한다.
- 보고서 작성에서는 결재선/보존년한/문서보안 필드를 기본으로 노출하지 않는다.

### 22.2 품질 기준

- API 호출이 없어야 한다.
- 새로고침해도 최근 작성 내역이 localStorage에 남아야 한다.
- 모바일에서도 깨지지 않아야 하지만, PC 기준으로 최적화한다.
- 모든 추천값에는 “최종 확인 필요” 안내를 포함한다.
- 사용자가 입력하지 않은 값은 프롬프트에서 “직접 입력”으로 표시한다.

## 23. 개발 시 주의사항

1. 결재선 추천은 확정값이 아니라 실무 참고값으로 표시한다.
2. 예시파일과 최신 규정집이 충돌할 경우 최신 규정집 기준을 우선한다.
3. 문서보안은 기본 2등급이나, 계약서/중요정보 포함 시 1등급 또는 대외비 검토 문구를 표시한다.
4. 첨부파일 추천은 문서 하단에 반드시 표시한다.
5. 결과물은 문서가 아니라 AI에게 붙여넣을 프롬프트다.
6. 외부 API 호출 코드를 만들지 않는다.
7. 개인정보, 계좌번호, 민감한 계약금액은 마스킹 권장 안내를 표시한다.
8. 입력값이 부족해도 프롬프트는 생성하되, 부족한 값은 “직접 입력”으로 처리한다.
